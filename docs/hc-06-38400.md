# Configure an HC-06 for 38400 baud

The project works at the HC-06 factory default of **9600 baud**. Change the
module to 38400 only when you deliberately select the matching firmware and
host settings. A baud mismatch does not damage the module, but it makes the
serial link appear completely dead.

This procedure is based on the manufacturer's
[HC-06 V2.0 manual](https://www.hc01.com/downloads/HC-06.pdf) and the
[older Linvor HC-06 datasheet](https://www.hc01.com/downloads/HC-06%20english%20datasheet.pdf).
Both specify 9600 8N1 as the default, `AT+BAUD6` for 38400, and persistent
storage of the selected baud. Modules sold under the same name may contain
different firmware. Stop if the replies do not match one of the documented
families below; do not guess commands from another module's guide.

## What you need

- the HC-06 removed from the robot link;
- a USB-to-TTL serial adapter with **3.3 V UART logic**;
- a supply suitable for the exact module or carrier board;
- Python 3 and pyserial (`python3 -m pip install pyserial`);
- the adapter's serial-device path.

The manufacturer's bare module uses 3.3 V UART levels and a 3.0–3.6 V supply;
its manual asks for at least 100 mA supply capacity. Some carrier boards add a
regulator, but their power-input rating is board-specific. Follow the marking
and schematic for the board in hand. Never infer 5 V tolerance from the
`HC-06` name.

Wire the isolated setup as follows:

| HC-06 | USB-TTL adapter / supply |
| --- | --- |
| VCC | verified module/carrier supply |
| GND | supply ground and adapter GND |
| TXD | adapter RXD |
| RXD | adapter TXD at 3.3 V logic |

Do not leave the Uno or motor board connected to these UART wires while
configuring the module.

## Step 1: find the adapter port

On macOS:

```sh
ls /dev/cu.*
```

Look for the USB-TTL adapter, commonly `/dev/cu.usbserial-...` or
`/dev/cu.usbmodem...`. This is the wired adapter port, not the Bluetooth
RFCOMM port.

## Step 2: enter HC-06 AT mode

1. Make sure no phone or computer is connected to the HC-06 over Bluetooth.
2. Power-cycle the module.
3. Leave it unconnected over Bluetooth. The manufacturer's firmware enters AT
   mode automatically when powered and not paired/connected; no KEY pin
   sequence is required.
4. Wait at least one second before the first command. The manuals also require
   about one second between commands.

LED patterns differ between carrier boards, so use an `OK` response—not the
LED—as proof of AT mode.

## Step 3: identify the firmware at 9600

Edit only the `PORT = ...` line below, then run the whole block:

```sh
python3 - <<'PY'
import time
import serial

PORT = "/dev/cu.usbserial-CHANGE-ME"

with serial.Serial(PORT, 9600, bytesize=8, parity="N", stopbits=1,
                   timeout=0.5, write_timeout=0.5) as link:
    for command in (b"AT", b"AT+VERSION"):
        link.reset_input_buffer()
        link.write(command)       # deliberately no CR or LF
        link.flush()
        time.sleep(1.2)
        print(command.decode(), "->", repr(link.read(64)))
PY
```

Proceed only if:

- `AT` returns `OK`; and
- `AT+VERSION` identifies the documented `hc01.comV2.0` family or a
  `LinvorV1.n` family.

Capitalization in the version response can vary between documented revisions.
The significant parts are `hc01.comV2.0` or `LinvorV1.`.

If `AT` returns nothing, do not send a baud-changing command. Work through the
troubleshooting table below first.

## Step 4: set 38400

Keep the terminal at 9600 and run this block once:

```sh
python3 - <<'PY'
import time
import serial

PORT = "/dev/cu.usbserial-CHANGE-ME"

with serial.Serial(PORT, 9600, bytesize=8, parity="N", stopbits=1,
                   timeout=0.5, write_timeout=0.5) as link:
    link.reset_input_buffer()
    link.write(b"AT+BAUD6")       # official HC-06 syntax; no CR or LF
    link.flush()
    time.sleep(1.2)
    print(repr(link.read(64)))
PY
```

The documented response is `OK38400`. The setting is retained after power is
removed.

If the response is different, stop. Record the exact bytes returned and locate
the manual for that firmware before trying anything else.

## Step 5: power-cycle and verify at 38400

Power-cycle the HC-06, keep it disconnected over Bluetooth, then run:

```sh
python3 - <<'PY'
import time
import serial

PORT = "/dev/cu.usbserial-CHANGE-ME"

with serial.Serial(PORT, 38400, bytesize=8, parity="N", stopbits=1,
                   timeout=0.5, write_timeout=0.5) as link:
    link.write(b"AT")
    link.flush()
    time.sleep(1.2)
    print(repr(link.read(64)))
PY
```

Do not reinstall the module until this returns `OK`.

## Step 6: select 38400 in this project

Add this line to the shared `[uno]` `build_flags` list in `platformio.ini` so
both deployable images use the selected baud:

```ini
-DROBOT_HOST_BAUD=38400UL
```

Then build the required image:

```sh
pio run -e robot
# or
pio run -e calibration
```

Use the same baud on the host:

```sh
warehouse-robot run --port /dev/cu.HC-06 --baud 38400
```

The HC-06, Uno firmware, and Python process must all use the same value. The
project does not send AT commands and cannot reconfigure the module over the
normal robot protocol.

## Return the module to 9600

Open the isolated adapter at its current 38400 baud and send `AT+BAUD4`. Expect
`OK9600`, power-cycle, then verify `AT` at 9600. Remove the 38400 override from
the shared `[uno]` flags to use the compiled 9600 fallback, or replace it
explicitly with:

```ini
-DROBOT_HOST_BAUD=9600UL
```

and run the host with `--baud 9600`.

## Troubleshooting and common misunderstandings

| Symptom or belief | What to do |
| --- | --- |
| `AT` has no reply | Confirm common ground and crossed TX/RX, disconnect every Bluetooth client, power-cycle, wait one second, and verify 9600 8N1. Confirm the adapter TX is 3.3 V logic. |
| The reply is readable only at another baud | That baud is the module's current UART setting. Identify the firmware there before changing it. |
| The module was changed and is now “lost” | Probe the documented PC-usable rates at or below 115200 using **only** `AT`: 9600, 38400, 19200, 57600, 115200, 4800, 2400, and 1200. Wait at least one second per attempt. The V2.0 manual lists still higher settings but says they require MCU-based access; do not try them casually. Do not send `AT+BAUD...` until `AT` and `AT+VERSION` work. |
| `AT` works but `AT+VERSION` is unknown | Treat the board as an undocumented variant. Save the exact reply and obtain its vendor manual. Do not assume the command or line-ending syntax in this guide. |
| A terminal sends `AT\r`, `AT\n`, or `AT\r\n` | The scripts above send the exact documented command bytes without a line ending. Some variants behave differently, which is another reason to identify the firmware before writing settings. |
| An online guide says `AT+BAUD=6`, `AT+UART...`, or requires a KEY sequence | Those forms are not the syntax in the two official HC-06 manuals linked above. Do not mix command sets. |
| Changing `--baud` should configure the module | It does not. `--baud` only changes the computer serial port. Configure the HC-06 separately, then make all three endpoints match. |
| The AT session worked at 9600, so robot data should work at 38400 | AT success proves only the baud used for that session. Step 5 is the required proof that the stored setting changed. |
| Reset should restore the factory baud | The official manuals document persistence but no hardware factory-reset procedure. A reset/restart is not evidence that 9600 was restored. |
| The Bluetooth settings page says “Disconnected” | RFCOMM serial links commonly open only while an application owns the serial port. Qualify the link with `warehouse-robot list-ports`, a successful port open, and a valid protocol-v3 HELLO/status exchange. |
| Text is garbled | One endpoint has the wrong baud, parity, or stop-bit setting. This project uses 8N1 at either 9600 or 38400. |

If none of the documented rates returns `OK`, stop rather than trying random
write commands. Recheck power and logic levels, then obtain the exact
carrier/module documentation or use the manufacturer's configuration hardware.
